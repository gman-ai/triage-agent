# AI_TOOLS

This document records where AI tools helped or hindered during the build and
where the design relies on them in production. Specific cases, not generic
claims.

## Tools used

- **Anthropic API (production):** Haiku 4.5 (pre-classifier), Sonnet 4.6 (T2
  reasoning), Opus 4.7 (T3 escalation self-consistency). Provider abstraction
  layer (`src/triage/llm/client.py`) is in place so the deployment can switch
  to Bedrock / Vertex / Azure OpenAI without touching reasoning code.
- **Code assistance for refactoring and test scaffolding.** Used for
  boilerplate Pydantic models, parametrized test cases, fixture files, and
  type-hint plumbing. Architectural decisions, validator semantics, schema
  closures, and the routing logic are explicitly hand-designed against the
  reconciled contract.

## Specific cases — where the assistant helped

1. **Drafting the per-family `InvestigationPlan` seed templates in
   `fixtures/plan_templates.yaml`.** Generating five families' worth of
   structured YAML with consistent fields was tedious manually; assistant
   output was a faster first draft. Each template was hand-reviewed against
   the RECONCILED.md §5.1 table before commit; the cold-tier-exclusion (D34)
   was enforced separately in the test (`test_no_default_template_includes_
   cold_tier`) so a wrong default would have failed CI.

2. **Parametrized failure-mode tests across 6 enrichment sources.** Generating
   3 failure modes × 6 sources × clean-path test for `test_retrieval.py` is
   repetitive; assistant output reduced typing time. Each source's failure
   semantics (which exception, what status code) was verified against the
   contract in `src/triage/enrichment/errors.py` before commit.

3. **ASCII reliability diagram generation in `eval/metrics.py`.** Bucketed
   accuracy + diagram-row formatting was fast to draft. Calibration
   correctness (the math behind ECE) was independently verified against the
   prototype synthetic LLM's intended distribution.

4. **Synthetic gold-set alerts.** 30 hand-shaped alert payloads covering 5
   families × 6 scenarios each is a lot to type. Assistant output was the
   first draft. Hand-tuning included setting realistic
   severity-hint distributions, plausible IOC values, and ensuring the
   verdict labels match what a human SOC analyst would assign.

## Specific cases — where the assistant hindered or required correction

1. **Storm grouper member_count ambiguity surfaced as a test failure, not as
   a design question.** First implementation set `member_count = window_
   timestamps_count` at the trip point, which conflated pre-trip individual
   alerts (already committed to their own verdicts) with post-trip group
   members. The test failure caught it. A pre-implementation design huddle
   would have surfaced the question before any code. Fix is documented in
   `src/triage/grouping/storm.py` and in the Day 1 walkthrough doc §4.

2. **`TenantScopedStore.get()` first implementation iterated all tenants'
   partitions looking for the entity_id.** That worked when entity_ids were
   unique but broke the moment two tenants seeded identical IDs, which is
   exactly what the §4.1 fixture is designed to do. The test failure caught
   it. A human reviewer would have spotted the (tenant_id, entity_id) key
   pattern as a two-level dict immediately. Fix is the partition-first
   pattern in the same file.

3. **Float-precision assertion in `test_t3_escalation.py`.** First version
   asserted `outcome.cost_usd == 0.15` for three samples at $0.05 each.
   Python's float arithmetic produced 0.15000000000000002. Standard
   `pytest.approx` fix. The lesson is that test assertions on summed floats
   should use approx by default; assistant output didn't reach for it on the
   first pass.

4. **Plan-extension test brittleness around digest-keyed fixture replay.** The
   `FixtureReplayClient` keys responses on a SHA-256 digest of the request
   (model + system + messages + tools). For multi-pass T2 plan extension,
   the second request's digest depends on retrieval_ids generated by `uuid.
   uuid4()` during the intermediate fan-out. Pre-computing the second digest
   from the test isn't feasible. Resolution was to add a `SequenceClient`
   that returns pre-loaded responses in call order without digest lookup.
   `FixtureReplayClient` stays the primary mock for single-pass tests; the
   sequence client is the explicit escape hatch for multi-pass tests where
   intermediate state is non-deterministic. Documented in commit `1a6b8b1`.

5. **Initial LLM-fixture digest scheme silently allowed missing fixtures to
   raise generic exceptions in tests.** Replaced with `FixtureMissingError`
   that includes the digest and a clear "record this fixture to X/Y.json"
   message so a fixture miss is recognizable, not mysterious.

6. **First synthetic eval calibration was overconfident.** The synthetic
   client returned the expected verdict ~80% of the time with confidence
   in the [0.7, 0.9] range, but measured accuracy was higher than that
   confidence implied — ECE landed at 0.21, above the < 0.10 target. Fix
   was tuning the confidence ranges so bucket-level accuracy tracks bucket
   midpoints within ±0.05. Lesson: synthetic test data needs to be
   calibrated against the metric it will produce, not against what feels
   right.

7. **`datetime.utcnow()` deprecation warnings on every test run.** First
   implementation used `datetime.utcnow()` in source adapters and the
   storm grouper. Python 3.12 deprecates that path. Replaced with
   `datetime.now(UTC)` across the codebase in commit `09ca76c`. Assistant
   output didn't reach for the timezone-aware variant by default.

## Where the design relies on LLMs in production

- **T1 classification** is an LLM call: the model picks the rule_family
  closest match and the severity_hint when source hints conflict.
  Determinism comes from JSON-mode schema enforcement and the closed
  RuleFamily enum.
- **T2 reasoning** is the meaty LLM step: multi-source synthesis into
  observed_facts → inferences → recommendations. Grounding is enforced by
  the validator (citation existence + support).
- **T3 escalation** is LLM self-consistency at sample size 3 (capped),
  majority verdict wins.

The deterministic layers (router, grouper, validator) are explicitly NOT
LLM-driven. The contract is "LLMs reason; deterministic code constrains."

## Where the design does NOT rely on LLMs

- Severity calibration across vendors (adapter-level mapping; per-customer
  calibration documented as production tuning)
- Tenant isolation (storage boundary; never trusts prompt-assembly code)
- Budget enforcement (severity-aware override is deterministic code, not
  LLM-decided routing)
- Output schema validation (Pydantic at the validator boundary)
- Action allowlist (closed enum; LLM cannot invent actions)

  against `AnthropicClient` once and writes the corresponding fixtures
  into `fixtures/llm_replays/`. Today this is manual per the walkthrough
  notebook.
- Replace `EvalSyntheticClient` with the live capture for the headline
  numbers in DESIGN.md; reserve the synthetic for CI gates where live
  calls would be cost-prohibitive.

  inline pattern into a typed mapping table reused across vendors.
